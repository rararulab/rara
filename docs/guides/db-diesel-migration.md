# sqlx → diesel Migration Strategy

Tracking epic: [#1702](https://github.com/rararulab/rara/issues/1702).
Feature branch: `feat/db-diesel`.

> **Status (2026-04-24)**: complete. `sqlx` is no longer a dependency of any
> rara crate; the workspace builds and tests against diesel 2.3 +
> diesel-async 0.8 + bb8 0.9. Local SQLite databases created under the old
> sqlx flow must be rebuilt once via `just migrate-reset` — diesel tracks
> applied migrations in `__diesel_schema_migrations` and does not read
> `_sqlx_migrations`.

This document is the single source of truth for how rara migrated its DB layer from sqlx 0.8 to diesel 2.3. Decisions below are **locked** — changes require a new RFC-level discussion on #1702, not an inline amendment.

## Scope

- Full migration of the rara DB layer from `sqlx` 0.8 to `diesel` 2.3. All-in, one-time. No dual-write period, no runtime feature flag, no sqlx-and-diesel coexistence beyond the transitional PRs in the stack.
- Trigger: `libsqlite3-sys` `links = "sqlite3"` collision with `boxlite` (see [#1696](https://github.com/rararulab/rara/issues/1696)). Cargo refuses to link two crates that claim the same native library. Moving off `sqlx` resolves this at the root rather than patching around it.
- This is a strategic refactor. It is *not* a minimum-diff workaround. Query-builder DSL adoption (below) is the cost we are explicitly paying to get long-term maintainability.

## Async strategy

- Use `diesel-async` + `bb8` connection pool. **Not** `tokio::task::spawn_blocking`.
- Rationale: rara is tokio-first. Wrapping a sync `diesel` pool inside `spawn_blocking` would add a blocking thread-pool hop to every query and would leak through cancellation semantics. `diesel-async` integrates directly with tokio and gives us a real async API surface.
- `bb8` (vs `deadpool` / `mobc`) is chosen because it is the tokio-native pool with the smallest surface area and aligns with the rest of the ecosystem (`tokio`, `hyper`, `axum`).

## Query style

- Full `diesel` DSL via `schema.rs` + the `table!` macro. **Not** `diesel::sql_query` as an escape hatch.
- Rationale: per explicit user directive — best practices over minimum-diff. `sql_query` would let us mechanically translate `sqlx::query!` call sites but would throw away the main reason to adopt diesel (type-checked queries, schema-aware joins). This is the primary driver behind the "expensive but one-time" cost profile.
- `sql_query` is permitted **only** in narrow cases where DSL coverage genuinely does not exist (e.g. FTS5 virtual-table operators, PG-specific `CREATE EXTENSION` bootstrap). Each such case requires a comment pointing back to this doc.

## Schema source of truth

- File: `crates/rara-model/src/schema.rs`.
- Generated by `diesel print-schema` against a freshly-migrated local DB.
- Regenerated whenever a migration changes the schema. The regenerated file is committed to the repo — `schema.rs` is not a build artifact.
- Rationale: matches `rara-model`'s role as the centralized schema/migration crate.

## Migration tool

- `diesel_migrations` (the embedded-at-compile-time macro variant), parallel to today's `sqlx::migrate!` usage in `rara-app`.
- Migrations are embedded into the binary at build time. Runtime does not read migration files from disk in production.

## Migration file format

- Under `crates/rara-model/migrations/`, each migration is a subdirectory (`YYYYMMDDHHMMSS_<name>/`) containing a `up.sql` + `down.sql` pair. This is diesel's canonical layout. The original flat `*.up.sql` / `*.down.sql` files from the sqlx era were restructured into subdirectories once at cutover; their contents were **not** modified (only the paths changed).
- Checksum semantics change (SQLx → diesel), so the `_sqlx_migrations` table is replaced by `__diesel_schema_migrations` on first boot post-migration. Local databases need a one-shot `just migrate-reset` to drop the old sqlx table and let diesel re-apply everything against its own tracking table.

## No schema redesign

- Migration is 1:1 functional. No tables are added, removed, or restructured as part of the sqlx → diesel work. Schema refactors, if any, land in separate PRs *after* the migration completes.
- Rationale: any schema churn mixed into this migration would make the diff unreviewable and make rollback impossible.

## Contributor setup

One-time local setup (added to the contributor docs in a later step of the stack):

```bash
cargo install diesel_cli --no-default-features --features sqlite,postgres
```

`diesel_cli` is required for `diesel print-schema` (regenerating `schema.rs`) and `diesel migration generate` (scaffolding new migration pairs). It is **not** a runtime dependency.

## Rollout

Stacked PRs off `feat/db-diesel` (see #1702 decomposition):

1. **#1703 (this PR)** — workspace deps + strategy doc. No code. No consumers. Zero behavioral change.
2. `schema.rs` generation + `diesel.toml` + `AGENT.md` update in `rara-model`.
3. Connection pool helper (`diesel-async` + `bb8`) in a shared location.
4. Per-crate repository migration (one PR per crate, smallest blast radius first).
5. Cutover: `sqlx::migrate!` → `diesel_migrations::embed_migrations!` in `rara-app` bootstrap.
6. Integration tests green on `testcontainers` (sqlite + postgres).
7. Final PR removes `sqlx` from workspace deps and from every crate `Cargo.toml`.

Final summary PR goes from `feat/db-diesel` → `main` for reviewer approval.

## What step 1 (this PR) explicitly does NOT do

- No Rust source changes (`**/*.rs` untouched).
- No `schema.rs` generation — that is step 2.
- No `diesel.toml` — that is step 2.
- No connection pool helper — that is step 3.
- No consumer crate adds `diesel` to its own `Cargo.toml` — workspace declaration only.
- No removal of `sqlx` anywhere — that is step 8.
- No `justfile` targets for `diesel-migrate-add` etc. — those ship when migration tooling actually shifts.
- No changes to files in `crates/rara-model/migrations/`.

## Version pinning decisions

Resolved on crates.io at PR creation time:

| Crate | Workspace constraint | Latest published | Notes |
|---|---|---|---|
| `diesel` | `^2.3` | `2.3.7` | Features used: `sqlite`, `postgres`, `chrono`, `uuid`, `serde_json`, `r2d2`. All verified present on `diesel = 2.3.7`. |
| `diesel-async` | `^0.8` | `0.8.0` | Features used: `sqlite`, `postgres`, `bb8`. **Deviation from the original ticket**: ticket suggested `0.5`; upstream has shipped through `0.8.0` and `0.5.x` is no longer the current line. Pinned to `0.8` to match current diesel `2.3`. |
| `diesel_migrations` | `^2.3` | `2.3.1` | Features used: `sqlite`, `postgres`. |

### Why these versions

- `diesel 2.3.x` is the current stable series. The `2.3` minor line has been stable since 2025 and includes the backend features we need (sqlite + postgres + JSON + r2d2 pool integration).
- `diesel-async 0.8.x` is the first series fully tracking `diesel 2.3`. Earlier 0.4/0.5 lines targeted `diesel 2.1`/`2.2` and would pin us to a stale core. The minor jump vs the ticket estimate is upstream churn, not a scope change.
- `diesel_migrations 2.3.x` tracks the diesel core version exactly — they are released in lockstep from the same repo. There is no other valid pairing.

### Bumping policy

- Bump `diesel` + `diesel_migrations` together. They live in the same repo and their versions are always in sync.
- Bump `diesel-async` whenever `diesel` major/minor bumps. Their compatibility matrix is tight — see upstream `README.md`.
- Never let any of the three diverge from the other two without explicitly testing the combination on CI.
