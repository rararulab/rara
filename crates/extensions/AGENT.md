# extensions — Agent Guidelines

## Purpose

Workspace group containing feature extensions that build on the kernel but are not part of the core agent loop.

## Sub-Crates

| Crate | Purpose |
|-------|---------|
| `rara-git` | Git operations library with SSH key management (ED25519 keypair generation, `GitRepo` abstraction) |
| `backend-admin` | Unified HTTP admin API (axum) aggregating all backend subsystems: sessions, agents, MCP, skills, settings, kernel info |

## Architecture

- `rara-git` is a standalone utility — no kernel dependency.
- `backend-admin` is a **facade** — routes delegate to kernel, sessions, MCP, and skills subsystems. It owns no business logic itself.

## Critical Invariants

- `backend-admin` state initialization order matters — `BackendState` must be constructed with valid service handles before routes are mounted.
- SSH keys generated by `rara-git` are stored via `rara-paths` — do NOT hardcode key paths.

## What NOT To Do

- Do NOT put business logic in `backend-admin` — it is a routing layer only. Logic belongs in the subsystems it delegates to.
- Do NOT add kernel dependencies to `rara-git` — it must remain a standalone utility.
- Do NOT add new top-level routes without registering them in the router aggregation in `backend-admin/src/lib.rs`.

## Dependencies

- **Downstream**: `app` and `server` crates consume `backend-admin`.
- **Upstream**: `backend-admin` depends on `rara-kernel`, `rara-sessions`, `rara-mcp`, `rara-skills`, `yunara-store`, `rara-domain-shared`. `rara-git` depends on `git2`, `ssh-key`, `rara-paths`.
